# 后台管理系统设计大纲（内部）

## 1. 设计目标
- 把当前“API 列表页”升级为真正的后台管理系统。
- 后台与代理客户端鉴权分离，后台采用独立账号密码 + session cookie。
- 先搭出可用后台骨架，再逐步填充统计、审计与运维能力。

## 2. 设计边界
- 保持现有代理能力不退化。
- 不把后台登录复用为客户端 Bearer token。
- 不引入前端构建链；首版以服务端渲染 + 原生 JS 为主。

## 3. 认证与会话
- 登录入口：`/login`
- 退出入口：`/logout`
- 会话方式：cookie + session id
- 后台账号来源：`config/admin_users.json`
- 主配置补充 session 参数：cookie 名称、secret、过期时间、滑动过期开关
- 账号字段建议：`username`、`password_hash`、`role`、`disabled`、`updated_at`

## 4. 页面结构
### 4.1 总览页
- 关键指标卡：请求量、成功率、失败率、当前在线目标、近 24h 消费
- 最近请求、近期告警、最近变更摘要

### 4.2 客户端管理
- 客户端列表
- 新增 / 编辑 / 删除
- 访问目标配置、说明、启用状态

### 4.3 模型费用
- 模型列表
- 单价维护
- 费用估算说明

### 4.4 消费统计
- 近 30 天日均柱状图
- 每小时 / 昨日 / 时间段汇总
- 按用户、按模型聚合

### 4.5 审计 / 日志
- 登录、修改客户端、修改费用、重载配置、失败操作日志

## 5. 路由草案
- `/login`
- `/admin`
- `/admin/overview`
- `/admin/clients`
- `/admin/model-costs`
- `/admin/usage`
- `/admin/audit`
- `/admin/api/*`

## 6. 数据模型草案
- `config/admin_users.json`：后台账号
- `config/clients.json`：客户端账号
- `config/model_costs.json`：模型单价
- `config/usage_events.jsonl`：消费事件
- `config/admin_audit.jsonl`：后台操作审计（建议新增）

## 7. 页面交互要求
- 左侧菜单 + 顶部状态栏 + 主内容区
- 列表页支持搜索、刷新、分页（分页可后置）
- 表单提交后页面局部刷新，不整页跳转
- 统计页支持时间范围选择

## 8. 接口分层
- 后台认证接口：登录、登出、当前会话、修改密码
- 后台管理接口：账号、客户端、费用、统计、审计
- 后台页面接口：HTML 页面渲染 / 静态资源

## 9. 里程碑
1. 登录与会话骨架
2. 后台布局与导航
3. 客户端管理页
4. 模型费用与统计页
5. 审计页与收尾优化

## 10. 暂不做
- 多租户
- 细粒度 RBAC
- 第三方 SSO
- 复杂权限继承

## 11. 核心流程
### 11.1 登录流程
1. 用户访问 `/login`。
2. 提交账号密码。
3. 服务端校验账号状态与密码哈希。
4. 登录成功后创建 session，重定向到 `/admin`。
5. 未登录或 session 过期时，后台中间件跳转回 `/login`。

### 11.2 页面加载流程
1. 进入 `/admin` 后加载总览页。
2. 左侧导航切换不同业务页。
3. 页面通过后台 API 拉取数据并局部刷新。
4. 刷新与操作成功后保留当前路由状态。

### 11.3 配置变更流程
1. 在客户端/费用页面提交修改。
2. 后台接口写入对应文件型数据源。
3. 记录审计日志。
4. 必要时触发运行时重新加载（如果变更影响后台展示或代理行为）。

### 11.4 审计流程
1. 所有关键变更记录审计事件。
2. 审计事件至少包含时间、操作者、动作、对象、结果。
3. 审计页支持按时间和对象过滤查看。

## 12. 页面级数据契约
### 12.1 总览页
- 需要：概览指标、最近请求摘要、告警摘要、最近变更摘要。

### 12.2 客户端管理页
- 需要：客户端名称、access key、允许目标、启用状态、备注、更新时间。

### 12.3 模型费用页
- 需要：模型名、输入单价、输出单价、缓存输入单价、最近更新时间。

### 12.4 消费统计页
- 需要：时间范围、按用户聚合、按模型聚合、小时/日桶、估算费用。

### 12.5 审计页
- 需要：动作、操作者、对象、结果、时间、详情。

## 13. 后台 API 约定
### 13.1 认证接口
- `POST /login`：提交账号密码，成功后设置 session cookie。
- `POST /logout`：销毁 session 并跳转登录页。
- `GET /admin/api/me`：返回当前登录账号与角色。

### 13.2 总览接口
- `GET /admin/api/overview`：返回首页关键指标、最近请求摘要、告警摘要、最近变更摘要。

### 13.3 客户端管理接口
- `GET /admin/api/clients`
- `POST /admin/api/clients`
- `PUT /admin/api/clients/{name}`
- `DELETE /admin/api/clients/{name}`

### 13.4 模型费用接口
- `GET /admin/api/model-costs`
- `PUT /admin/api/model-costs/{model}`
- `DELETE /admin/api/model-costs/{model}`

### 13.5 消费统计接口
- `GET /admin/api/usage/summary`
- `GET /admin/api/usage/aggregate`
- `GET /admin/api/usage/events`

### 13.6 审计接口
- `GET /admin/api/audit`
- `GET /admin/api/audit/{id}`（可选）

## 14. 账号与权限策略
- 首版可先支持单一角色 `admin`，但数据结构预留 `role` 字段。
- 管理账号默认禁用公开注册，仅允许离线配置或人工创建。
- 密码必须哈希存储，不可明文保存。
- 账号状态至少支持：启用 / 禁用 / 重置密码。

## 15. 会话与安全策略
- Cookie 建议开启：`HttpOnly`、`SameSite=Lax`、`Secure`（HTTPS 环境）。
- Session 超时与滑动过期可配置。
- 对所有写接口建议附加 CSRF 防护：同站请求策略 + CSRF token（二选一或组合）。
- 登录失败次数应有基础限制，避免暴力猜测。

## 16. 错误与反馈规范
- 后台 API 统一返回 JSON 错误：`code`、`message`、`request_id`（如可用）。
- 前端以 toast / banner 方式展示错误，不直接展示堆栈。
- 删除、重载、保存等高风险操作需要二次确认。

## 17. 交付与实现顺序
1. `/login` + session 基础
2. `/admin` 布局 + 导航壳
3. 总览页
4. 客户端管理页
5. 模型费用页
6. 消费统计页
7. 审计页

## 18. 验收标准（设计阶段）
- 任何实现者看到该文档，应能直接拆出路由、接口、数据文件和页面任务。
- 文档不依赖口头补充即可判断首版后台的范围。
- 设计清楚标明“先做什么 / 暂不做什么 / 为什么”。
