## 🚨 红线清单（绝对禁止！）

以下命令在 33.110 上**永远不要执行**，任何一条都会导致生产数据丢失：

| # | 禁止命令 | 后果 |
|---|---|---|
| ❌ | `git clean -fdx` / `git clean -fd` | **删除 `data/llms-proxy.db`、`config/config.json`、`logs/*`** ← 历史事故的元凶 |
| ❌ | `rm -rf data/` 或 `rm -rf config/` | 数据彻底丢失 |
| ❌ | `docker compose down -v` | `-v` 会删除 volume |
| ❌ | `git checkout .` 在有未跟踪文件时强行切换分支 | 可能丢弃配置 |
| ❌ | 不跑 `scripts/deploy.sh` 就直接 `git reset --hard` | 虽然不删 untracked，但缺少备份链路 |

**唯一正确的发布方式**：`bash scripts/deploy.sh`（脚本已内置保护）

## 📋 2026-04-17 事故复盘

| 项 | 内容 |
|---|---|
| 现象 | 部署后 `config/config.json` 和 `data/llms-proxy.db` 被清空，所有目标/客户端/admin 密码丢失 |
| 根因 | `data/` 当时**未加入 `.gitignore`**，是 untracked 状态；有人（或脚本）执行了 `git clean -fdx` 类命令，把所有 untracked 文件一扫而空 |
| 为何 `git reset --hard` 本身不是元凶 | `git reset --hard` 只重置**已跟踪**文件，不动 untracked。但 `git clean -fdx` 会 |
| 修复措施 | ① `.gitignore` 加入 `data/`；② `deploy.sh` 加发布锁、预检、持久备份、健康检查、自动打印回滚命令；③ 增加本红线清单 |
| 残留风险 | 如果有人手工在服务器上跑 `git clean -fdx`，即使在 `.gitignore` 里 `-x` 标志依然会删。**必须靠红线清单+培训防范** |

### 方式一：使用部署脚本（唯一推荐）

```bash
bash scripts/deploy.sh
```

脚本会自动完成：
1. **发布锁**：防止并发部署
2. **预检**：检查 data/llms-proxy.db、config.json 存在；未提交改动需人工确认
3. **持久备份**：备份到 `/DATA/Backups/llms-proxy/<时间戳>/`（保留最近 20 份）
4. **探测分支**：自动识别 `origin/main` 或 `origin/master`
5. **绝不 git clean**：避免删除 untracked 的 data/
6. **git reset --hard** → 恢复 config/ data/
7. **重建容器** → **30 秒内健康检查**
8. **失败时打印回滚命令**（不自动回滚，由人工决策）

### 方式二：手动操作（仅在脚本坏了时使用）

```bash
# ① 备份运行时数据（绝对不能跳过！）
BACKUP="/DATA/Backups/llms-proxy/manual-$(date +%Y%m%d%H%M%S)"
mkdir -p "$BACKUP"
cp -a config/ "$BACKUP/config"
cp -a data/   "$BACKUP/data"
git rev-parse HEAD > "$BACKUP/previous-commit.txt"
echo "已备份到 $BACKUP"

# ② 拉取最新代码（不要加 git clean！）
git fetch origin
git reset --hard origin/main   # 若 master 分支则替换

# ③ 恢复运行时数据（双保险）
cp -a "$BACKUP/config"/. config/
cp -a "$BACKUP/data"/.   data/

# ④ 确保目录权限
mkdir -p config data logs
chmod 777 config data logs
[ -f config/config.json ] && chmod 666 config/config.json

# ⑤ 重建并启动容器
docker compose up --build -d

# ⑥ 验证
sleep 5
curl -sf http://localhost:8000/healthz && echo OK
docker compose ps
ls -la config/config.json data/llms-proxy.db
```

### 回滚（发布失败时）

脚本在健康检查失败时会自动打印完整回滚命令。手动回滚示例：

```bash
LATEST_BACKUP=$(ls -1t /DATA/Backups/llms-proxy/ | head -1)
BACKUP_DIR="/DATA/Backups/llms-proxy/$LATEST_BACKUP"
PREV_COMMIT=$(cat "$BACKUP_DIR/previous-commit.txt")

cd /DATA/AppData/llms_proxy
docker compose down
cp -a "$BACKUP_DIR/config"/. config/
cp -a "$BACKUP_DIR/data"/.   data/
git reset --hard "$PREV_COMMIT"
docker compose up --build -d
```

## 2. 部署路径

```

├── docker-compose.yml             # Compose 定义
├── config/                        # 配置目录 → 容器内 /etc/llms-proxy
│   └── config.json                # 运行时配置
├── data/                          # 数据目录 → 容器内 /var/lib/llms-proxy
│   └── llms-proxy.db              # bbolt 数据库（客户端、用量、审计等）
├── logs/                          # 日志目录 → 容器内 /var/log/llms-proxy
│   ├── access.log
│   └── error.log
└── deploy/docker/Dockerfile       # 构建用 Dockerfile
```

## 3. Docker Compose 配置

```yaml
services:
  llms-proxy:
    container_name: llms-proxy
    build:
      context: .
      dockerfile: deploy/docker/Dockerfile
    image: llms-proxy:latest
    restart: unless-stopped
    environment:
      - SERVER_BIND=0.0.0.0:8000
      - LOG_LEVEL=info
    ports:
      - 8000:8000
    volumes:
      - ./config:/etc/llms-proxy        # 可读写，admin 保存配置需要写入权限
      - ./data:/var/lib/llms-proxy
      - ./logs:/var/log/llms-proxy
```

### 关键说明

- **config 目录不能挂载为 `:ro`**。容器进程以 `llmsproxy`（uid=100）用户运行，admin 界面保存目标/配置时需要写入 `config.json`。如果挂载为只读，运行时修改只在内存生效，重启后丢失，并报错：
  ```
  config applied at runtime but failed to persist to disk; changes will be lost on restart
  ```
- 服务直接通过局域网 IP `192.168.33.110:8000` 访问，**不经过 nginx 反向代理**。

## 4. 文件权限要求

容器进程以非 root 用户运行：

```
uid=100(llmsproxy) gid=101(llmsproxy)
```

宿主机上对应的挂载目录必须确保该 uid 有读写权限：

```bash
# config 目录：属主设为容器用户，保证 admin 可以持久化配置
chown -R 100:101 /DATA/AppData/llms_proxy/config/

# data 和 logs 目录：需要读写权限（默认 777 已满足）
# 如果权限不够，执行：
chown -R 100:101 /DATA/AppData/llms_proxy/data/
chown -R 100:101 /DATA/AppData/llms_proxy/logs/
```

> **注意**：uid 100 在宿主机上可能显示为 `dhcpcd` 等无关用户名，这是正常的，不影响功能。

## 8. 故障排查清单

| 现象 | 可能原因 | 排查命令 |
|---|---|---|
| 外部无法访问 8000 端口 | 容器端口映射丢失（被接入了错误的网络） | `docker inspect llms-proxy --format '{{json .HostConfig.PortBindings}}'`，应返回含 `8000` 的映射 |
| admin 保存配置报错 "failed to persist to disk" | config 目录无写入权限或挂载为 `:ro` | `docker exec llms-proxy touch /etc/llms-proxy/test && echo OK`，失败则修复权限 |
| 容器启动后立即退出 | 配置文件格式错误 | `docker compose logs` 查看具体错误 |
| 容器名冲突无法创建 | 旧容器残留 | `docker rm -f llms-proxy && docker compose up -d` |

## 9. 注意事项

1. **该机器的 Docker 使用 containerd image store**（`io.containerd.snapshotter.v1`），`docker save` 默认输出 OCI 格式。如需导出镜像给 QNAP 等设备，需使用 `skopeo` 转换为 Docker 传统格式，详见 `AGENTS.md` 中的说明。
2. 该主机上同时运行 nginx 反向代理容器（服务 chat/dify/termix 等域名），llms-proxy **不需要**通过该 nginx 转发，保持独立的端口映射即可。
3. `admin_session.secret` 当前为开发默认值，生产环境应更换为强随机字符串（至少 32 字符）。
4. 镜像体积约 34 MB（Alpine + 静态编译 Go 二进制），资源消耗极低。
